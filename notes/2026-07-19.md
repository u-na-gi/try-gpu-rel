# 2026-07-19 — Triton Tutorial 01: Vector Addition

## 環境(初回セットアップ、動作確認済み)

- GPU: RTX 5060 Ti 16GB(Blackwell, **sm_120**)、SM 36基、warp 32、L2 32MiB
- 理論メモリ帯域: **448 GB/s**(GDDR7 28Gbps × 128bit)
- WSL2(Ubuntu)、Python 3.14 / uv 管理
- triton 3.7.1 / torch 2.13.0+cu130(arch list に sm_120 同梱を確認)
- nvcc なし。Triton は ptxas 同梱 + ドライバのみで動く(CUDA Toolkit 不要)

## やったこと

1. `add_kernel`(vector addition)を実装 → torch の `x + y` と突き合わせ、
   **max difference 0.0** で一致(要素ごと独立の演算なので浮動小数点まで完全一致。
   行列積では加算順の差で誤差が出るはず → allclose 比較になる)
2. `triton.testing.do_bench` + `perf_report` でサイズ 2^12〜2^27 をベンチ

## 結果と分析

飽和スループット **393 GB/s = 理論値の 87.7%**。帯域律速カーネルとしてはほぼ上限
(DRAM は実効 85〜90% が実用上の天井)。Triton と torch は全サイズで数%差 = 互角。

ベンチ曲線は3幕構成で読める:

- **〜13万要素: 固定費の世界。** サイズ2倍で GB/s もほぼ2倍 = 実行時間がほぼ一定。
  逆算するとカーネル起動の固定費 ≈ **4.2 µs**(12B × 4096 ÷ 11.6 GB/s)
- **26万要素(256 program)あたりで転換。** 36 SM に十分な供給が回り始める
- **以降: 帯域律速の世界。** 393 GB/s に漸近して飽和

GB/s の計算: 要素1個 = load 4B ×2 + store 4B = **12 バイト/要素**。
`gbps = 3 * numel * element_size / time`

公式ドキュメントの図版(飽和 ~1686 GB/s、データセンター級 GPU)との比較:

- 天井は 4.3 倍違うが**曲線の形は同一**(3幕構成はハード非依存)
- 小サイズ(4096要素)ではむしろ 5060 Ti が速い(11.6 vs 8.0 GB/s)。
  起動固定費は GPU のグレードで安くならない
- 同じ Triton コードが両方のハードで理論値の8割超を出す = 性能可搬性

## 概念の整理(自分の言葉で)

- **カーネル** = GPU で実行される関数。特殊なのは呼び出し方(1回の起動で大量の
  インスタンスが同じコードを一斉実行 = SPMD)
- **program** = カーネルの実行インスタンス。OS アナロジーでは
  カーネル=実行ファイル、**program=プロセス**、その下の実スレッド=スレッド。
  program 間は隔離(DRAM + atomic でのみ連携)、program 内スレッドは共有(SRAM)
- **グリッド** = program を何個・何軸で並べるかのタプル。`cdiv` で切り上げて多めに
  起動し、カーネル側の `mask = offsets < n_elements` で範囲外を守る(セットで定石)
- **offsets** = `block_start + tl.arange(0, BLOCK_SIZE)`。担当要素番号のベクトル。
  `x_ptr + offsets` で初めてポインタになる
- カーネル内にループは書かない。外側ループ→グリッド、内側ループ→ベクトル演算に
  吸収され、後者はコンパイラが実スレッド(warp 単位)へ割り付ける
- **共有カウンタへの `+= 1` は加算消失する**(read-modify-write の競合)。
  `tl.atomic_add` で不可分にする。Rust の `fetch_add` と同じ理屈
- カーネル内の `print` は `tl.device_print`(第1引数はラベル文字列必須)。
  スレッドごとに出力されるので同一行が大量に出る = program の下の実スレッドが見える

## ハマったこと

- `torch.cuda.is_available` を `()` なしで if に使うと常に真(関数オブジェクトの
